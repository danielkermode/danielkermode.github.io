Problem Solving

Friday, 2016-04-15

Getting Blocked

Goodness, there have been many times that I have been blocked on a piece of code. Generally I try to stay calm and go through the problem step-by-step and try to work out the control flow of the program, and console.log everything I can along the way (I think this is the most popular method of debugging for Javascript developers). Here are three general things that seem to block me on a regular basis:

  • 1. Management of state (in JS apps Redux can handle this quite nicely).
  • 2. Understanding exactly how a library/framework functions.
  • 3. Building something that is cross-browser and easy to deploy.

Management of state proved to be an issue in the racecar app I made this week. I thought state would be simple since it was such a small program, but it was tricky to handle different states in conjunction with messages from the server (a socket.io server) in such a way that the app worked with many different "paths" (for example the user exiting a game then challenging another player). This is related to the second point, in that I initially had issues understanding exactly how to use socket.io and wasn't aware of the best way to use the API. I still certainly don't claim expertise with socket.io, but it got easier as I kept researching and googling better ways to do things.

By far the most frustrating and time-consuming problem with the racecar app was not knowing to call document.removeEventListener after the game had finished. It meant the React UI component I was rendering for the game didn't unmount from the DOM properly, and caused annoying state issues. In hindsight, I think I could attach the event listener to the component rather than the global document, but my current implementation is working fine. It's funny how small oversights like these can often consume disproportionate amounts of time while coding.

I haven't found cross browser issues to be too bad, but they can be a bit of a pain. My first implementation of the racecar app checked the "code" property of a "keydown" event, which apparently doesn't exist in Safari and IE. So I did some googling and discovered that a better way to get the letter of the key pressed (in all browsers) is to use String.fromCharCode.

Deployment for the racecar app was (thankfully) very easy with Heroku. Since it was a simple standalone node.js app, I just followed the (good) documentation on Heroku's website for Node and my app can be deployed with git push heroku master. However with databases and other dependencies, I've found that deployment can sometimes be a headache.

Some JS Array Methods

Array methods can remove verbose for loops and make your code more expressive.

.map() is one of the most useful. Basically what it does is call a function on each element in an array, and return a new array with each element adjusted by the function. It doesn't mutate the existing array. So for example, if you had a function "dyeRed(carpet)" and you had an array with a bunch of blue carpets, you could run blueCarpets.map(dyeRed); and you will receive an array of red carpets (however, the blueCarpets array will be the same).

.filter() also applies a function to each element in an array, but will return the element if the function gives a "true" and will remove the element if the function gives a "false". Again, it doesn't mutate the existing array. So say you have an array of people and you had a function "isGeek(person)", you could run people.filter(isGeek); and you'd get the geeks returned back to you, but your original "people" array would look the same.

.reduce() is a bit more complicated. It still applies a function to every array element, but the function's first argument will be the "previous" value of the value accumulated so far, and the second argument will be the current element. As well as taking in this function, it takes in an optional starting value. Then it will go through each of the elements, successively applying the function, and return a single value (again without modifying the array). The most simple example of how it works is with an "add(a, b)" function that adds two numbers.
Calling [1, 2, 3].reduce(add, 10); will successively add each value of the array to the starting value of 10. So you'll get 10 + 1 + 2 + 3 = 16. The function doesn't have to add, it could do any operation successively on each element of the array. A more abstract example: say you have an array of colours, and a function "mixColour(a, b)". Calling colours.reduce(mixColour); would give you a successive mix of all the colours. So you'd probably get brown, or maybe white if it was light (I'm not really a colour expert).
